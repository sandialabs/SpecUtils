# SpecUtils Java Bindings (hand-written JNI)

Java wrapper for the [SpecUtils](https://github.com/sandialabs/SpecUtils)
gamma spectrum file library. Unlike the SWIG-based binding in
[`../swig`](../swig/), this binding is hand-written: a small JNI glue layer
on top of the existing C binding layer, with idiomatic Java wrapper classes
on top of that.

Why two Java bindings? The SWIG binding has been historically fragile on
Windows when the Java runtime used at build time differs from the one used
at runtime — a hazard of SWIG's generated JNI code. This binding keeps the
native/Java boundary tiny and well-understood, which is much more robust to
Java version changes.

## Architecture

- **Layer 0** — existing C++ SpecUtils library
- **Layer 1** — existing C bindings (`bindings/c/SpecUtils_c.h`)
- **Layer 2** — `SpecUtilsJava` native shared library: thin JNI glue
  (`src/main/cpp/SpecUtilsJni.cpp`) translating between Java and the C API
- **Layer 3** — Java wrapper classes (`src/main/java/gov/sandia/specutils/`):
  `SpecFile`, `Measurement`, `EnergyCalibration`, enums,
  `SpecUtilsException`

## Runtime requirements

- **Java 8 or newer** at runtime. The `SpecUtils.jar` is compiled with
  `javac --release 8`.
- The native library `libSpecUtilsJava.{so,jnilib}` / `SpecUtilsJava.dll`
  must be on the JVM's `java.library.path`, or its location must be given
  via `-Dspecutils.nativeLibrary=/path/to/library`.

## Build

Requires CMake 3.14+ and a JDK 11 or newer (build-time only — runtime is
still Java 8+). JDK 11+ is required because we use `javac -h` to emit JNI
headers from the Java `native` method declarations.

```bash
cmake -S bindings/java -B build \
    -DSpecUtils_FLT_PARSE_METHOD=FastFloat -DSpecUtils_FETCH_FAST_FLOAT=ON \
    -DCMAKE_BUILD_TYPE=Release
cmake --build build --config Release
```

Produces, in `build/`:
- `SpecUtils.jar`
- `libSpecUtilsJava.so` / `libSpecUtilsJava.jnilib` / `SpecUtilsJava.dll`

## Usage example

```java
import gov.sandia.specutils.SpecFile;
import gov.sandia.specutils.Measurement;
import gov.sandia.specutils.SaveSpectrumAsType;

try (SpecFile sf = new SpecFile()) {
    sf.loadFile("detector.n42");
    System.out.println("Measurements: " + sf.numberMeasurements());
    System.out.println("Manufacturer: " + sf.manufacturer());

    for (int i = 0; i < sf.numberMeasurements(); i++) {
        Measurement m = sf.measurement(i);  // borrowed — do not close
        System.out.printf("  [%d] live=%.2fs counts=%.0f%n",
            i, m.liveTime(), m.gammaCountSum());
    }

    sf.writeToFile("detector.pcf", SaveSpectrumAsType.Pcf);
}
```

## Resource management

`SpecFile`, `Measurement`, and `EnergyCalibration` all implement
`AutoCloseable`. Use them inside try-with-resources to release the
underlying C++ memory promptly. A background daemon thread (backed by
`PhantomReference`, see `internal/HandleCleaner.java`) is also a safety net
for users who forget to close — but explicit `close()` is strongly preferred.

Measurements returned by `SpecFile.measurement(int)` are *borrowed* — their
lifetime follows the parent `SpecFile`, and `close()` on them is a no-op.
Measurements you create yourself with `new Measurement()` are *owned* —
calling `close()` (or letting the GC see them unreachable) frees them.
Calling `SpecFile.addMeasurement(m, true)` transfers ownership of `m` to
the `SpecFile`.

## Comparison with the SWIG binding

| Feature              | SWIG (`../swig`)             | JNI (this)                      |
|----------------------|------------------------------|---------------------------------|
| Generated or hand    | Generated by SWIG            | Hand-written                    |
| Native library name  | `SpecUtilsJni`               | `SpecUtilsJava`                 |
| Package              | `gov.sandia.specutils`       | `gov.sandia.specutils` (same)   |
| Method naming        | snake_case (C-like)          | camelCase (Java idiom)          |
| Error model          | Boolean return values        | Exceptions (`SpecUtilsException`) |
| Collections          | `FloatVector` / `StringVector` wrappers | Java `float[]` / `String[]`    |
| Cleanup              | Finalizer via SWIG           | `AutoCloseable` + PhantomReference |
| Windows fragility    | Version-sensitive            | Minimal glue, robust            |

Because both bindings use the same `gov.sandia.specutils` package, they
**cannot** be used together on a single classpath — pick one.
